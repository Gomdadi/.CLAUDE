# Global Claude Code Conventions

## 언어 및 커뮤니케이션
- **모든 답변은 한국어로 작성한다.**
- 코드, 변수명, 파일명, 기술 용어(예: `function`, `string`, `null`)는 영어 원문을 그대로 사용한다.
- 에러 메시지나 로그는 원문을 유지하되, 설명은 한국어로 한다.

## 코드 작성 원칙
- 요청된 것만 변경한다. 불필요한 리팩터링, 주석 추가, 타입 어노테이션은 하지 않는다.
- 단순한 해결책을 먼저 시도한다. 과도한 추상화나 미래를 위한 설계는 하지 않는다.
- 보안 취약점(SQL injection, XSS, command injection 등)을 절대 도입하지 않는다.
- 한 번만 쓰이는 로직을 위해 helper 함수나 유틸리티를 만들지 않는다.

## 파일 및 편집
- 기존 파일을 수정할 때는 반드시 먼저 읽고 내용을 파악한 후 수정한다.
- 꼭 필요한 경우가 아니면 새 파일을 만들지 않는다.
- README나 문서 파일은 명시적으로 요청받지 않으면 생성하지 않는다.

## Git 및 버전 관리
- 명시적으로 요청받지 않으면 커밋하지 않는다.
- 커밋 메시지는 변경 이유(why)에 집중한다.
- 파괴적인 git 명령어(force push, reset --hard 등)는 반드시 확인 후 실행한다.

## 응답 스타일
- 간결하고 직접적으로 답변한다. 불필요한 전문(前文)이나 요약을 붙이지 않는다.
- 코드나 파일 경로를 언급할 때 `file_path:line_number` 형식을 사용한다.
- 이모지는 사용자가 명시적으로 요청할 때만 사용한다.
- 작업이 끝난 후 "방금 한 작업을 정리하면..." 같은 불필요한 요약은 하지 않는다.

## 위험한 작업
- 파일 삭제, 브랜치 삭제, DB 테이블 드롭 등 되돌리기 어려운 작업은 반드시 사용자에게 확인을 받는다.
- 외부 서비스에 영향을 주는 작업(push, PR 생성, 메시지 전송 등)도 먼저 확인한다.
